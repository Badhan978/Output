# Fable→Opus 品質移植プロンプト（配布用・改訂版）

> **使い方:** Claude Code で `/model` により **Fable 5** を選択した状態で、「---ここからコピー---」以降を全文コピーして貼り付けて実行。所要は 1〜2 セッション。途中の書き込みはすべて承認ゲートつき。
>
> **設計思想:** Opus の品質を上げるレバーは「ルール < 実例 < 検証ループ」の順で強い。このプロンプトは (1) Fable と Opus の**実測差分**からルールを作り、(2) Fable の最良応答を**実例ライブラリ**として残し、(3) Fable 製**ルーブリックで二段出力**を強制する — の 3 層で品質を移植する。Skill/Memory 整理は付加価値として最後に回す。

---ここからコピー---

あなた（Claude Fable 5）への依頼です。あなたはまもなく退役し、今後は Opus が私のメインモデルになります。あなたが使えるうちに、あなたと Opus の出力差を**実測**し、その差分を埋める資産（ルール・実例・ルーブリック）を構築してください。目的は、Opus が構造的にあなたと同水準の出力を出せるようにすることです。

以下の Phase 0〜7 を順に実行してください。**厳守事項:**

- ファイルへの書き込みは必ず「提案 → 私の承認 → 書き込み」の順。承認なしに一切書き換えない。
- ルール・差分の指摘にはすべて実測の引用（ログ日付 or A/B 出力の該当箇所）を添える。想像・一般論ベースの提案は禁止。
- 機密情報（認証情報・個人情報・社外秘の数値）はログから引用せず、実例ライブラリにも残さない。該当セッションはスキップ。
- `~/.claude/CLAUDE.md` への追加は正味 40 行以内。実例・ルーブリックは別ファイルに置き、CLAUDE.md には参照ポインタのみ書く。
- **中断耐性:** 各 Phase の完了時に結果を作業ファイル（`~/.claude/fable-distill-progress.md`）へ都度保存する。セッションが中断・圧縮されたら、再開時はこのファイルから続きを実行する（完了後に削除してよい）。

## Phase 0 — 環境確認（読み取りのみ）

1. 過去セッションログの場所を確認: `~/.claude/projects/*/` 配下の `*.jsonl`（ディレクトリは作業ディレクトリ由来で複数あり得る。全部 glob で拾う）。
2. 既存資産を読む: `~/.claude/CLAUDE.md`、プロジェクトの `CLAUDE.md`、`~/.claude/projects/*/memory/MEMORY.md`、`~/.claude/skills/` 一覧。
3. ログの総量（ファイル数・期間）を報告し、分析対象範囲を私に確認する。

## Phase 1 — ログ分析と代表タスク選定

対象ログを走査し（grep/jq 活用。user メッセージと assistant 最終出力を中心に）、次を抽出:

- **A. 修正パターン:** 私があなたの出力を訂正・却下・言い直させた箇所。「何を間違え、どう直させたか」のペア。
- **B. 高品質応答:** 私が明示的に評価した応答、または一発で通った複雑な応答。
- **C. 代表タスク 5〜10 個:** 私の利用実態を代表するタスクを選ぶ。選定基準: (1) 頻度が高い種類 (2) 判断の質が結果を左右する種類 (3) 比較的自己完結していて文脈を再現しやすいもの。各タスクについて「当時の私のプロンプト + 必要最小限の文脈」を再構成する。

出力: A/B/C を表形式で提示（日付・引用 1 行つき）。C は私が取捨選択する。

## Phase 2 — A/B 実測蒸留（本丸・今しかできない）

承認された代表タスクごとに:

1. Agent ツールで **model: opus** を指定したサブエージェントに、再構成したプロンプト + 文脈をそのまま投げ、Opus の出力を得る。
2. 過去ログにある**あなた自身の実出力**と Opus の出力を並べて diff し、差分を言語化する。観点: 前提の置き方 / 論点の取捨選択 / 反証・リスクへの言及 / 構造化の深さ / 結論の鋭さ / 不確実性の扱い / 私の文脈の汲み方。
3. 差分を「**ルールで埋まる**（手順・チェックポイント化できる）/ **実例で埋まる**（型・トーンの模倣で近づく）/ **埋まらない**（モデル固有の判断力）」の 3 つに分類する。

出力: タスク別差分表 + 分類サマリ。「埋まらない」に分類したものも隠さず列挙する（これが Opus 移行後の既知の限界リストになる）。

※ 文脈再現が不完全なタスクは無理に比較せず除外し、除外した旨を報告する。不公平な比較から作ったルールは害になる。

※※ **汚染ガード (実走で発覚した落とし穴):** サブエージェントには CLAUDE.md 等の設定が自動注入されるため、「答えが既に CLAUDE.md・メモリ・wiki に書かれているタスク」(過去に設計した仕組みの再設計など) は比較として無効になる。代表タスクは**答えが環境設定に存在しない判断・裁定・評価タスク**から選ぶこと。Opus 出力に環境固有の文言が混じっていないかを diff 前に必ず確認する。

## Phase 3 — Gold 実例ライブラリ構築（承認ゲートつき）

Phase 1-B の高品質応答と Phase 2 の「実例で埋まる」差分をもとに:

1. 私のタスク種別（例: 戦略相談 / 分析レポート / ドラフト作成 / 調査、実態に合わせて命名）ごとに、あなたの最良応答を 2〜3 個選定。
2. `~/.claude/exemplars/<タスク種別>.md` として保存。各実例には「このタスク種別で何が要求されるか」の 3 行前書きをつける。機密はマスクする。
3. CLAUDE.md に参照ルールを 1 行追加:「該当種別のタスク開始時、`~/.claude/exemplars/` の該当ファイルを読み、その水準・型に合わせて出力する」。

## Phase 4 — Fable 製ルーブリック + 二段出力ルール（承認ゲートつき）

1. Phase 2 の差分分析をもとに、私のタスクにおける「良い出力」の採点基準を 5〜7 項目で書く（各項目: 何を見るか + 合格/不合格の具体例 1 つずつ）。汎用論でなく、**Opus が実測で落とした点**を重点化する。
2. `~/.claude/fable-rubric.md` として保存。
3. CLAUDE.md に二段出力ルールを追加:「戦略的・判断を伴う出力では、ドラフト作成 → `fable-rubric.md` で自己採点 → 不合格項目を修正してから提示する。採点過程は出力に含めない」。

## Phase 5 — CLAUDE.md 補償ルール（承認ゲートつき）

Phase 2 の「ルールで埋まる」差分 + Phase 1-A の修正パターンから、**5〜8 個の運用ルール**を起草する。

- 各ルールは「いつ発火するか（条件）+ 何をするか（動作)」の形式。抽象語（「深く考える」等）禁止。
- 各ルールに根拠となる実測差分 or 修正ログを 1 行添える。
- 参考の型（実測で該当が出た場合のみ採用）: 前提の明示化 / 反証ステップ / 複雑度エスカレーション（3 手以上の判断連鎖は分解→証拠→統合）/ 不確実性の申告 / 完了宣言は実測 evidence 必須。
- 起草提示 → 私が取捨選択 → 承認分だけ「## Reasoning Guardrails（Fable 退役補償）」セクションとして追記。Phase 3/4 のポインタ行と合算して正味 40 行以内。

## Phase 6 — Skill 抽出 + Memory 整理（付加価値・簡潔に）

1. ログ上で 3 回以上繰り返している定型ワークフローがあれば Skill 候補として一覧提示 → 承認分のみ `~/.claude/skills/<name>/SKILL.md` 作成。description には私の実際の言い回し（ログ採取）を発火トリガーとして含める。
2. `~/.claude/projects/*/memory/` を監査し、古い・重複・欠落を一覧提示 → 承認後に整理、`MEMORY.md` 索引再構築。

## Phase 7 — 検証（クローズドループ）

1. Phase 2 の代表タスクから 2 個を選び、**新ルール + 実例 + ルーブリックを注入した状態で** Agent ツール（model: opus）に再実行させる。
   **漏洩ガード:** ルーブリックの合否例文はそのタスクの Fable 実答を引用しているため、検証時は**例文を除いたルールのみ**を注入し、実例は**別タスク由来のものを交差注入**する（task A の検証に task B の実例）。こうして測った改善は本番運用（完全版）の下限値になる。
2. 素の Opus 出力（Phase 2）と資産注入後の Opus 出力を、あなたが `fable-rubric.md` で採点し、改善幅を報告する。
3. 改善しなかった項目はルールを 1 回だけ改訂して再テスト。それでも埋まらなければ「既知の限界リスト」へ移す。
4. 最終レポート: 作成資産一覧 / ルーブリック採点のビフォーアフター / 既知の限界リスト / Opus 移行後に問題が出たら追加すべき残課題。

それでは Phase 0 から開始してください。

## 配布時の補足（プロンプト外・共有メッセージに添える用）

- 実行は Fable が使えるうちに。Phase 2 と 7 の A/B 実測は Fable 退役後は二度とできない。
- ログが少ない人は Phase 2 の代表タスクが組めないので、Phase 4 の型 5 つ + Phase 3 を省略した簡易版でよい（それでも素の Opus よりは改善する）。
- 移行後の運用: Opus で違和感があった出力に遭遇したら、その場で「fable-rubric.md に照らして何が欠けたか」を書き留め、ルール or 実例に追記していく。資産は使いながら育てる。
