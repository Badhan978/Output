4. 試すアングル — 種・テスト期間・打ち出しスケジュール

選定原理（実測由来）: CV は Imp でなく intent から出る。「コンテンツの続き = Audible」のゼロ距離接続が成立するほど上位。種は随時追記。
アングル種（ペルソナ × 接続）テスト期間うまくいった場合の打ち出し
ビジネス書・耳活 主軸積読サラリーマン（28-38・通勤40-60分）。「その積読、耳なら通勤で月3冊」— 要約で興味喚起 → 全編は耳で、の最短導線7/16〜31（cohort1）8月ロットの主軸に複製（同 playbook 10体上限 → バリアント展開）
怪談・朗読 対抗深夜の怪談リスナー（20-40）。音声で物語を聴く習慣が形成済み。「完全版（プロ朗読1時間）を無料で」。江戸の闇の当たり型・資産を転用7/16〜31（cohort1）耳活型と成績比較 → 8月ロットの配分を決定
話題小説・ミステリー 第2波読書に戻りたい30代女性。ドラマ化・映画化の原作需要。「あの話題作、家事しながら聴けます」8月（2週目ロット〜）週50体ロットに追加投入
朝活・習慣術 第2波自己変革願望層（20-30代）。「朝の30分、スマホの代わりに聴く」。耳活の姉妹型で playbook 共用8月（2週目ロット〜）週50体ロットに追加投入
お金・投資リテラシー 第2波NISA・家計見直し層（25-45）。マネー本要約 → 「通勤中にお金の勉強」。金融系高単価案件へのミックス布石8月（2週目ロット〜）追加投入 + 商材単価ミックス検証
歴史・時代小説 継続現行江戸クラスタ（0223/0193）。唯一 CV 実績のある接続 — baseline として維持継続中導線改修 + Story 20日化で設計値到達を実証（CTR 検証の母艦）

5-1. 走る前に先に解くべき課題点
項目解決策
Meta API 兵站（最大の単一障害点）API投稿はできない→Reels=API投稿・Stories＝手動投稿。
ASP の量産許容初期 = 江戸裏話（承認済みメディア）発行済みの Audible リンクを新規アカで共用 → アカが育ち次第（目安フォロワー3k = ASP 閾値）各アカ側でリンク発行し CV を分散。※共用期間は acc/ロット単位の CV 帰属なし → kill 条件③はクリック代理で運用、リンク切替後に CV 判定復帰
端末準備（1端末最大5アカウント — 方針転換 7/9: 同 playbook ロットは BAN リスク同一のため同居可、コスト効率優先）①台数 = 合計30台（7月分 10〜15台を即日発注、8月に追加購入。owner: Badhan — 調達先・単価確定→発注）②1端末 = 1ロット（同一 playbook 5アカ）固定 — 端末境界 = ロット隔離境界に一致させ BAN 切り分けを単純化 ③セットアップ分担 = Badhan・駿冴（+Zach は初期セットアップのみ）
全体のロジック計画が一気通貫で回る論理の事前固め = 3つの噛み合わせ確認: ①数値チェーン（Imp→Stories→Click→CV が前提レートで閉じるか）→ 計画表で消化済み ②ワークフロー12工程がチェーンのどの数値を動かすか（どの工程が落ちるとどの数値が死ぬか）→ 週次ワークフロー整合で消化済み ③週次判定ループ（計測→kill/scale→翌週計画）の閉じ → ロット kill 条件の最小数値セット確定（7/9・週次課題検知システム行参照）で閉了。3点とも解消
v5 が何かの課題に見舞われるか大量のPlaybookを追加するから、そのエラー。市場分析がどこまでワークするのか問題
フォロワー非依存導線（7月300 CV の成否を握る）標準形 = 三層搭載: ①Reel CTA『プロフから』固定文言 ②bio に A8 直リンク ③ハイライトに 6/23 最終Frameテンプレ常設。リンクを配置別（bio/ハイライト）に発行し初週ロットで実測 → CTA 文言のみ A/B。canary = プロフ来訪率 ≥3%。※bio 直リンクの規約可否は ASP 照会に追加
週次の課題検知システム（アングルXの継続バー含む）ロット kill 条件（最小セット）: ①初速 = 投稿開始2週でロット view 中央値 15k/本未満 → playbook kill（中央値=単発バズ除外。設計 66.7k/本の約22%水準）②導線 = 3週目にリンククリックが中央値 35件/acc/週 と ロット合計 35×体数/週 の両方割れ → kill、片方割れは導線修理1回→翌週再判定（設計 71件/acc/週の50%線）③CV = リンク稼働後3週でロット合計 CV 0 → kill（CV リードタイム約1ヶ月織り込み。既存アカの7日ルールとは別）※15k・35件は既存アカ実測からの外挿 — 初回ロット2週実測で1回較正 ④フォロワー到達ペース（watch 指標・kill ではない）= 週次でロット中央値フォロワーを計測、目安 375/週（8週で 3k = per-acc リンク発行と F1 複利の前提）。2週連続で半分以下なら共用リンク期間の延長を計画に反映 + 導線見直し
Stories 手動投稿の兵站確定（70体で約47件/日、150体で約100件/日。分担と Zach 参加の可否、Stories API 対応可否）①Lala が『リンクスタンプの API 付与可否』を 7/11 までに実確認（投稿可否でなくスタンプ可否が肝）②手動前提: 70体 = 約47件/日 ≈ 2h/日、早川メイン（日次4-5h 枠内）・バックアップ Lala ③150体 = 約100件/日 ≈ 5h/日 → 2〜3人分担で1.5〜2h/人 = 許容範囲 ④Zach 不参加（7/9 Q2 Recap のコア4人エグゼキューション外し lock に整合）※1件2〜3分は仮置き — 初週に早川実測で置き直し
Playbook 化への受け渡しロジック（生成担当から Tomo に何を渡すか）渡すもの: ・SessionLog ・コード類
v5 統合のリスク管理（ミス生成・バグによる無駄な API 費用を発生させない検証手順）生成費用を中央管理（利益がメインファクター化）: ①各バッチ前に『そのバッチで変えるもの』を事前設定（例: CTA を〇〇に変更 / テーマ変更）— 変更要求は全て中央集権的に集約し、意思決定後に生成を走らせる ②新規 Playbook は過去のミス生成要因の有無を Check → テスト生成1本 → 通過後に本走
